## 1. Аналіз Single Responsibility Principle (SRP)

Принцип єдиної відповідальності стверджує, що кожен клас повинен мати лише одну причину для зміни.

### 1.1. Приклади дотримання SRP



* **Клас `FactAttribute`**: 
    Його єдина відповідальність — маркування методу як тесту та зберігання базових метаданих (`DisplayName`, `Skip`). Він не займається пошуком тестів або їх виконанням.
* **Клас `Assert`**: 
    Кожен метод всередині (наприклад, `Equal`, `NotNull`) відповідає тільки за порівняння значень та генерацію виключення у разі невідповідності. Логіка виводу результату в консоль чи файл винесена в інші модулі.

### 1.2. Приклади порушення SRP
* **Клас `TestInvoker`**: 
    У деяких реалізаціях цей клас одночасно відповідає за створення екземпляра тестового класу, виклик методу через рефлексію та перехоплення помилок. Це робить клас складним ("God Object"), оскільки зміна в будь-якому з цих процесів вимагає модифікації одного й того самого файлу.

---

## 1.3. Аналіз Open/Closed Principle (OCP)

Принцип відкритості/закритості стверджує, що програмні сутності мають бути відкриті для розширення, але закриті для модифікації.

### 2.1. Приклади дотримання OCP



* **Механізм `DataAttribute`**: 
    Це класичний приклад OCP. Ядро xUnit вміє працювати з базовим класом `DataAttribute`. Якщо розробнику потрібно брати дані з JSON, Excel або бази даних, він просто створює новий клас (наприклад, `JsonDataAttribute`), не змінюючи вихідний код самого xUnit.
* **Інтерфейс `IMessageSink`**: 
    Дозволяє розширювати способи обробки повідомлень про результати тестів. Користувачі можуть додавати власні логери, реалізуючи цей інтерфейс, без переписування внутрішньої логіки раннера.

### 2.2. Приклади порушення OCP
* **Внутрішні перевірки статусів**: 
    У застарілих частинах коду, що відповідають за консольний вивід, іноді використовуються конструкції `switch` або `if-else` для обробки різних типів результатів. При додаванні нового статусу (наприклад, "Inconclusive") доводиться вносити зміни безпосередньо в логіку виводу.

---

## 3. Висновки по архітектурі
Проект xUnit демонструє високу якість проектування. Завдяки розділенню відповідальностей (SRP) код залишається читабельним, а використання абстракцій замість конкретних реалізацій (OCP) робить фреймворк найбільш гнучким інструментом у своїй категорії.
